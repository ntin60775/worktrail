# worktrail v2 — Дизайн системы

## Главный принцип

**Разработчик не оператор. Агент — единственный оператор системы.**

Разработчик просто работает: создаёт ветки, пишет код или текст, делает коммиты.
Knowledge-слой накапливается сам, как побочный продукт. Максимум участия человека —
ответить «да/нет» на уточняющий вопрос агента.

## Универсальность

Система не привязана к языку, стеку или типу проекта. Работает для Python, Rust,
Go, 1С, личного дневника, исследования, финансов — для любого git-репозитория.

Всё кодо-специфичное вынесено в pluggable адаптеры. Ядро оперирует универсальными
понятиями: задача, контракт, критерий успеха, верификация, решение.

## Бутстрап

**Одна команда. Один навык.**

1. Разработчик клонирует репо навыка
2. Говорит агенту: «Установи worktrail глобально»
3. Агент: копирует SKILL.md, ставит git-хуки (`git config --global core.hooksPath`),
   прописывает managed-блок в AGENTS.md, устанавливает зависимости
4. Всё. Ни `init`, ни `install apply`, ни проектной установки

Managed-блок в AGENTS.md — только роутинг. Он говорит агенту: «Если ты в
git-репозитории, активируй навык worktrail. Определи режим: исполнение или ревью.»

## Хранение данных

**Git-notes — единственное хранилище.**

Никакого SQLite, никакой файловой структуры `knowledge/`. Основные типы записей:

| Тип | Что хранит |
|-----|-----------|
| `contract` | Контракт задачи: ID, summary, scope, success_criteria, verification methods |
| `decision` | Архитектурное решение: rationale, alternatives, привязка к файлу:строкам |
| `spec` | Инварианты, привязанные к scope или файлу |
| `review_package` | Итоговый пакет для ревью: контракт + VRR + decisions + границы |
| `review_result` | Вердикт ревью |

Плюс **VRR-логи** (Verification Run Record) — JSONL временно на диске в
затронутой области, после завершения — итоговый VRR в git-note, черновики
удаляются.

## Жизненный цикл задачи

### Старт

```
Пользователь: «Сделай интеграцию с бухгалтерией, ERP-4521»
Агент:       resolve_context() → определяет задачу
             plan → контракт (YAML) → git-note
Пользователь: «Да, верно» (или корректирует)
```

### Исполнение

```
Агент:       работает по контракту
             record_decision() → git-note с привязкой к файлу:строкам
             record_progress() → structured footer в commit
             run_verification() → VRR (через адаптер под проект)
Пользователь: пишет код / текст, коммитит как обычно
```

### Завершение

```
Агент:       finalize()
             → собирает review_package (контракт + VRR + decisions + границы)
             → git-note
             → derive_time() по git-логу
             → build_report() → Markdown для человека
```

### Ревью

Новая сессия, тот же навык, режим review.

```
Пользователь: «Проведи ревью ERP-4521 командой экспертов»
Агент:       resolve_context() → review_package
             → определяет профиль проекта → набор экспертов
             → параллельно: 5 саб-агентов по осям (контракт, код, решения, границы, VRR)
             → единый отчёт: вердикт, блокеры, предупреждения
             → git-note с review_result
```

## Адаптеры верификации

Pluggable модули. Агент не знает, что под капотом. Вызывает `run_verification(method)` → получает VRR.

| Адаптер | Для чего |
|---------|---------|
| `pytest` | Python-проекты |
| `1c_test` | 1С-тестовые модули |
| `manual` | Чеклист с да/нет в терминале |
| `shell` | Произвольный скрипт |
| `none` | Проекты без верификации (дневник) |

Новый адаптер добавляется без изменения ядра — модуль с интерфейсом `run(task_id) → VRR`.

## Профили проекта

Влияют только на состав экспертной панели при ревью:

| Профиль | Эксперты |
|---------|---------|
| `generic` (Python/Rust/Go) | Контракт, Код, Решения, Границы, VRR |
| `1c` | Контракт, Код(.bsl), Метаданные, Решения, Границы, VRR |
| `diary` | Контракт, Содержание, VRR |
| `research` | Контракт, Источники, VRR |

## Миграция старых знаний

**Read-only просмотрщик `knowledge/tasks/`**. Читает `task.md` + `registry.md`,
выдаёт сводку по запросу. Worktrail v1 (`.worktrail/runtime.db`) не трогаем.

Никакой миграции данных. Старые проекты: агент обнаруживает `knowledge/` при
`resolve_context()`, читает как дополнительный источник контекста, но пишет
только в git-notes.

## Архитектура проекта

```
worktrail/                        ← один репозиторий, один глобальный навык
├── skill.md                      ← контракт навыка (оба режима)
├── agents.md.block               ← managed-блок для AGENTS.md
│
├── core/                         ← универсальное ядро
│   ├── contract.py               ← схема контракта
│   ├── git_notes.py              ← read/write/list
│   ├── time.py                   ← derive_time из git-log
│   ├── report.py                 ← build_report → Markdown
│   └── context.py                ← resolve_context
│
├── adapters/                     ← pluggable верификация
│   ├── base.py                   ← интерфейс
│   ├── pytest.py
│   ├── manual.py
│   ├── shell.py
│   └── none.py
│
├── executor/                     ← режим «исполнитель»
│   └── workflow.py               ← record_decision, finalize, VRR-цикл
│
├── reviewer/                     ← режим «ревьюер»
│   └── audit.py                  ← загрузка review_package → эксперты → отчёт
│
├── hooks/                        ← глобальные git-хуки
│   ├── prepare-commit-msg
│   ├── post-commit
│   └── post-checkout
│
└── archive/                      ← просмотрщик старых знаний
    └── tck_reader.py             ← read-only для knowledge/tasks/
```

## Что решили отложить

- **Derived time vs реальное время** — пока считаем по коммитам
- **Расширенные адаптеры** (Excel, screenshot-diff) — архитектура готова, конкретные адаптеры позже
- **Внешний docs hosting** — не в v1
